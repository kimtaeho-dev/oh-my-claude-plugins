# 프로젝트 지식 체계 — 산출물 예시

> 이 문서는 Skill 실행 결과로 생성되는 문서의 **완성된 예시**를 보여준다.
> 모든 예시는 **Planning Poker** 프로젝트 기준으로 통일되어 있다.

---

## 예시 프로젝트: Planning Poker

- **개요**: Jira 연동 Planning Poker 웹 앱. 실시간 투표 → 결과 공개 → Jira 이슈에 스토리 포인트 반영.
- **스택**: Next.js + WebRTC (FE 중심, 스택 유형 B)
- **도메인**: poker (투표/결과), room (방 생성/관리), jira (Jira 연동)

### POLICIES.md (poker 도메인)

```markdown
# 투표 (poker) 도메인 - 서비스 정책서

> 최종 갱신: 2025-06-15

---

## poker.투표

### Validation 규칙

#### POL-001: 최소 참가자 수
- **규칙**: 참가자가 2명 이상 모이고 전원이 카드를 선택해야 결과가 공개된다
- **위반 시**: 대기 화면이 유지되며 투표를 진행할 수 없다
- **비즈니스 배경**: Planning Poker는 팀 추정 기법이므로 최소 2인이 필요하다

### 설정값

#### POL-002: 투표 카드 옵션
- **규칙**: 피보나치 수열 기반 카드(1, 2, 3, 5, 8, 13, 21)와 특수 옵션 불확실(?), 휴식(☕)을 선택할 수 있다
- **비즈니스 의미**: 피보나치 수열은 작업 규모가 커질수록 추정 불확실성이 증가하는 특성을 반영한다
- **변경 시 영향**: 카드를 추가/제거하면 통계 계산 로직(평균, 최빈값)에 영향을 준다

### 에러 시나리오

#### ERR-001: 방을 찾을 수 없음
- **메시지**: "방을 찾을 수 없습니다"
- **발생 상황**: 초대 링크를 클릭했지만 해당 방이 이미 종료되었거나 존재하지 않을 때
- **사용자 영향**: 빈 화면 대신 에러 안내와 함께 홈 화면 링크가 표시된다

#### ERR-002: 연결 끊김
```

```markdown
## 투표

### 이 기능은 무엇인가
참가자가 현재 논의 중인 이슈에 대해 스토리 포인트 카드를 선택하는 기능이다. 모든 참가자가 투표를 완료하면 자동으로 결과 공개 단계로 넘어간다.

### 사용자 시나리오
1. 호스트가 이슈를 선택하면 모든 참가자에게 카드 선택 화면이 표시된다
2. 각 참가자가 카드를 선택한다 (선택 전까지 다른 사람의 카드는 뒤집혀 있다)
3. 전원이 선택을 완료하면 2초 카운트다운이 시작된다
4. 카운트다운이 끝나면 전원의 카드가 동시에 공개된다

### 왜 이렇게 동작하는가
- **카드를 숨기는 이유**: 다른 사람의 선택에 영향받지 않고 독립적으로 추정하기 위함 (앵커링 효과 방지)
- **2초 카운트다운**: 즉시 공개하면 마지막 투표자에게 부담이 없고, 너무 길면 흐름이 끊긴다
- **피보나치 수열 채택**: 작업 규모가 커질수록 추정 불확실성이 증가하는 특성을 카드 간격으로 표현

### 주요 정책
- POL-001: 최소 2명 참가 필수 (상세: POLICIES.md)
- POL-002: 피보나치 + 특수 카드 (상세: POLICIES.md)

### 에러 상황
- ERR-001: 방을 찾을 수 없음 — 홈에서 새 방 생성 또는 호스트에게 새 링크 요청
- ERR-002: 연결 끊김 — 자동 재접속 대기, 30초 이상 시 새로고침

- 단계(Phase): 투표 라운드의 현재 상태 (상세: GLOSSARY.md)

## 방 생성 마법사

### 이 기능은 무엇인가
Planning Poker 세션을 시작하기 위한 3단계 설정 화면이다. Jira 인증 → 닉네임 설정 → Epic 선택 순서로 진행한다.

### 사용자 시나리오
1. 사용자가 Jira 접속 정보(도메인, 토큰)를 입력한다
2. 시스템이 Jira에 접속하여 인증을 확인한다
3. 사용자가 닉네임을 설정한다
4. 사용자가 추정할 Epic의 키를 입력한다
5. 시스템이 해당 Epic의 하위 이슈 목록을 불러온다
6. 사용자가 "방 만들기"를 누르면 세션이 시작되고 초대 링크를 공유할 수 있는 대기 화면으로 이동한다

### 왜 이렇게 동작하는가
- **3단계 분리 이유**: Jira 인증 실패를 첫 단계에서 즉시 알려주고, 무거운 이슈 목록 로드는 마지막에 수행하여 각 단계의 성공/실패를 명확히 알 수 있게 한다
- **Jira Cloud/Server 자동 판별**: 이메일 입력 여부로 Cloud와 Server를 구분하므로, 사용자가 인증 타입을 직접 선택할 필요가 없다
- **인증 정보 저장**: 브라우저에 저장하여 재방문 시 입력을 생략한다 [수동 추가]

### 주요 정책
- POL-001 (jira): Jira 인증 필수 (상세: jira/POLICIES.md)
- POL-002 (jira): 이슈 최대 100건 (상세: jira/POLICIES.md)
- POL-001 (room): 방장만 설정 변경 가능 (상세: room/POLICIES.md)
```
