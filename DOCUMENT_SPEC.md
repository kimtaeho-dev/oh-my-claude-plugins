# 프로젝트 지식 문서 스펙 (Document Spec)

> 이 파일은 .claude/docs/ 하위 문서를 생성하거나 갱신할 때 참조하는 **형식 정의서**이다.
> Claude Code는 문서를 생성할 때 반드시 이 스펙의 형식을 따라야 한다.

---

## 공통 규칙

- 모든 문서 상단에 `> 최종 갱신: {YYYY-MM-DD}` 포함
- 기능 ID 형식: `{app}.{기능명_스네이크}` (예: `backoffice.포지션_생성_수정`)
- 동일 기능 ID가 DOMAIN_MAP.md, FEATURES.md, SERVICE_FLOW.md 간에 일관되게 사용되어야 함
- 영향 관계는 반드시 양방향(impacts / affected_by)으로 기술
- 코드에서 확인되지 않는 관계는 `[미확인]`으로 표기하고 추측하지 않음
- 언어: 도메인명·기능명은 한글, 코드 경로·엔티티명·기술 용어는 영문

---

## DOMAIN_MAP.md

### 역할
프로젝트 전체 도메인을 한눈에 보여주는 최상위 맵. "이 기능을 수정하면 어디에 영향이 가는가?"를 즉시 판단하는 근거.

### 형식

```markdown
# Domain Map

> 최종 갱신: {YYYY-MM-DD}
> 생성 기준: 코드베이스 자동 분석 + 수동 보정

## 도메인 맵

### {도메인 한글명} ({domain_english_name})

| 기능 ID | 영향을 주는 곳 → | ← 영향을 받는 곳 |
|---|---|---|
| {app}.{기능명_스네이크} | → {app}.{기능_ID}, {app}.{기능_ID} | ← {app}.{기능_ID} |

(도메인 수만큼 반복)

---

## 교차 도메인 영향 관계 (Cross-Domain Impact)

| 출발 | 도착 | 영향 유형 | 설명 |
|---|---|---|---|
| {app}.{기능_ID} | {app}.{기능_ID} | {데이터 의존 / 권한 변경 / 콘텐츠 동기화 / 이벤트 트리거 / 상태 전파} | {한 줄 설명} |
```

### 작성 규칙
- 기능 ID 열은 반드시 `{app}.{기능명_스네이크}` 형식을 사용하라. FEATURES.md의 섹션 헤딩과 완전히 동일한 문자열이어야 한다.
- "영향을 주는 곳 →" / "← 영향을 받는 곳" 열에서 참조하는 기능도 동일한 기능 ID 형식을 사용하라.
- "영향을 주는 곳 →" 열: 이 기능의 산출물이나 상태 변경이 트리거가 되어 다른 기능에 영향을 미치는 경우
- "← 영향을 받는 곳" 열: 다른 기능의 변경에 의해 이 기능의 동작이 달라지는 경우
- 영향이 없으면 `(없음)` 명시
- 교차 도메인 영향 관계 테이블은 도메인 경계를 넘는 관계만 기록 (같은 도메인 내 관계는 제외)
- 영향 유형은 다음 중 선택: 데이터 의존, 권한 변경, 콘텐츠 동기화, 이벤트 트리거, 상태 전파

## CODE_STRUCTURE.md

### 역할
코드베이스의 물리적 디렉토리 구조와 논리적 도메인을 매핑. Claude Code가 "이 도메인의 코드는 어디에 있는가?"를 즉시 파악하는 용도.

### 형식

```markdown
# Code Structure

> 최종 갱신: {YYYY-MM-DD}

## 프로젝트 루트 구조
(tree 명령 결과를 2depth까지 표시)

## 디렉토리 ↔ 도메인 매핑

| 디렉토리 경로 | 도메인 | 앱 | 비고 |
|---|---|---|---|
| `{path}` | {domain} | {app 또는 '공통'} | {필요시 설명} |

## 공유 모듈

| 모듈 경로 | 설명 | 사용 도메인 |
|---|---|---|
| `{path}` | {역할 설명} | {사용하는 도메인 목록} |

```

## DEPENDENCY_MATRIX.md

### 역할
전체 도메인 간 의존 관계를 매트릭스로 시각화. 리팩토링 시 영향 범위를 한눈에 파악하는 용도.

### 형식

```markdown
# Dependency Matrix

> 최종 갱신: {YYYY-MM-DD}
> 행(→) = 영향을 주는 도메인, 열(←) = 영향을 받는 도메인
> ●: 강한 의존 (데이터/로직 직접 참조)
> ○: 약한 의존 (이벤트/알림 수준)
> -: 의존 없음

|  | {domain_1} | {domain_2} | {domain_3} | ... |
|---|---|---|---|---|
| **{domain_1}** | - | {●/○/-} | {●/○/-} | ... |
| **{domain_2}** | {●/○/-} | - | {●/○/-} | ... |

## 의존 상세

### {domain_A} → {domain_B} ({●/○})
- {의존 관계 설명 1}
- {의존 관계 설명 2}
- 공유 엔티티: {엔티티 목록}
- 의존 방향 근거: {import 경로, 이벤트 구독 등 코드 근거}

(●/○ 인 관계마다 반복)
```

### 작성 규칙
- ● (강한 의존): 도메인 A의 코드가 도메인 B의 엔티티/서비스를 직접 import하거나, DB 테이블을 직접 참조
- ○ (약한 의존): 이벤트/메시지 큐를 통한 간접 연결, 또는 공유 인터페이스를 통한 느슨한 결합
- "의존 상세" 섹션에 반드시 코드 수준 근거(import 경로, 이벤트명 등)를 포함

## SERVICE_FLOW.md

### 역할
주요 유저 시나리오를 기능 흐름으로 기술. Claude Code가 "이 흐름에서 내가 수정하는 부분의 전후 맥락"을 파악하는 용도.

### 형식

```markdown
# Service Flows

> 최종 갱신: {YYYY-MM-DD}

## FLOW-{NNN}: {흐름 제목}

### 참여 도메인
{domain_1}, {domain_2}, ...

### 흐름

1. **[{app}] {단계명}**
   - 도메인: {domain}
   - 기능: {app}.{기능_ID}
   - 산출물: {엔티티} (status: {상태})
   - 트리거: {이전 단계에서 어떤 조건이 충족될 때}
   - 영향: → {다음에 영향받는 기능}

2. **[{app}] {단계명}**
   ...

### 흐름 내 수정 영향 분석

| 수정 지점 | 영향받는 후속 단계 | 확인 사항 |
|---|---|---|
| Step {N} ({설명}) | Step {M}, {K} | {구체적 확인 포인트} |
```

### 작성 규칙
- 흐름 ID는 `FLOW-{NNN}` 형식으로 순번 부여
- 각 단계의 기능 ID는 DOMAIN_MAP.md / FEATURES.md와 동일하게 사용
- "흐름 내 수정 영향 분석" 테이블 필수 포함
- 최소 3개 이상의 주요 흐름을 식별 (코드베이스에서 가장 핵심적인 유저 시나리오 기준)

## domains/{domain-name}/FEATURES.md

### 역할
도메인 내 각 기능의 상세 영향 관계와 코드 위치를 매핑. 수정 전 영향 범위 확인과 수정 후 검증의 기준 문서.

### 형식

```markdown
# {도메인 한글명} 도메인 - 기능 목록

> 최종 갱신: {YYYY-MM-DD}

---

## {app}.{기능_ID}

- **설명**: {기능 설명 1~2문장}
- **코드 위치**:
  - FE: `{경로}` (해당하는 경우)
  - BE: `{경로}` (해당하는 경우)
  - API: `{METHOD} {endpoint}` (해당하는 경우)
- **주요 엔티티**: {Entity1}, {Entity2}
- **영향을 줌 (impacts)**:
  - `{app}.{기능_ID}` — {영향 설명}
    - 트리거: {어떤 조건에서 영향이 발생하는가}
    - 영향 범위: {구체적으로 어떤 데이터/화면/로직이 영향받는가}
  - (영향을 주는 기능이 없으면 `(없음)` 명시)
- **영향을 받음 (affected_by)**:
  - `{app}.{기능_ID}` — {영향 설명}
    - 의존 필드: {어떤 필드/데이터에 의존하는가}
  - (영향을 받는 기능이 없으면 `(없음 — 최상위 생성 기능)` 등으로 명시)
- **변경 시 체크리스트**:
  - [ ] {조건} 시 → {확인 사항}
  - [ ] {조건} 시 → {확인 사항}

---

(기능 수만큼 반복)
```

### 작성 규칙
- 코드 위치는 실제 코드베이스 탐색으로 확인한 경로만 기재
- impacts의 트리거/영향 범위는 코드에서 확인 가능한 수준으로 구체적으로 작성
- 체크리스트는 "이 기능을 수정했을 때 반드시 확인해야 할 것"을 기능의 영향 관계로부터 도출

## domains/{domain-name}/ENTITIES.md

### 형식

```markdown
# {도메인 한글명} 도메인 - 엔티티 정의

> 최종 갱신: {YYYY-MM-DD}

## {EntityName}

- **설명**: {엔티티 설명}
- **코드 위치**: `{파일 경로}`
- **주요 필드**:
  | 필드명 | 타입 | 설명 | 참조 도메인 |
  |---|---|---|---|
  | {field} | {type} | {설명} | {다른 도메인에서 참조하면 명시, 아니면 '-'} |
- **상태 전이** (해당하는 경우):
  - {status_A} → {status_B}: {전이 조건}
  - {status_B} → {status_C}: {전이 조건}
- **관계**:
  - {EntityName} → {OtherEntity}: {관계 설명} ({1:N, N:M 등})
```

### 작성 규칙
- 필드 목록은 DB 스키마 또는 모델 클래스에서 확인한 것만 기재
- "참조 도메인" 열: 이 필드가 다른 도메인의 엔티티에서 읽히거나 변경 트리거가 되는 경우 해당 도메인 명시
- 상태 전이가 있는 엔티티는 반드시 전이 다이어그램 기술 (다른 도메인에 영향을 주는 상태 변경이 핵심)

```markdown
# {도메인 한글명} 도메인 - 변경 이력

> 최종 갱신: {YYYY-MM-DD}

## {YYYY-MM-DD}: {변경 제목}

- **변경 내용**: {무엇이 변경되었는가}
- **영향받은 기능**: {app}.{기능_ID}, {app}.{기능_ID}
- **교차 도메인 영향**: {있으면 기술, 없으면 '없음'}
- **문서 갱신**: FEATURES.md {갱신함/미갱신}, DOMAIN_MAP.md {갱신함/미갱신}
```
